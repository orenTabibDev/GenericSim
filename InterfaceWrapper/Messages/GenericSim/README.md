# GenericSim

A WPF message simulator (MBT side) generated by **InterfaceWrapper**.
It consumes the native `WrapperFromCToSharp.dll` to build and parse messages.

## Build order

1. Build `WrapperFromCToSharp.vcxproj` for **x64** (produces the DLL).
2. Build/run `GenericSim.csproj` (x64). The DLL is copied next to the executable.

## Features

- Message list with per-message editable field grid (Offset/Field/Type/Size/Value).
- Array fields (e.g. bit/discrete arrays) appear as rows inside the fields grid
  at their real offset; the arrow expander opens a nested cells table under the
  row where every element can be edited, and collapses it when not needed. The
  element count box sets how many elements are sent (the count field is synced).
- Auto sequence / timestamp / CRC toggles.
- Live HEX preview built through the native convert functions.
- Selectable transport: UDP, TCP (client or server) or RS232 serial.
- Transport settings in the top bar (ports, IP, TCP mode, COM port, baud rate, parity, stop bits).
- Periodic send (interval + count), send history, NIRON→MBT monitor and traffic log.
- Export/Import message field values as JSON.
- Scenario tab: drag messages from 'All Messages' into an ordered step list,
  set per-step attributes (one time / periodic, start time, interval, max count)
  and run all steps in parallel. Step fields support arrays: expand an array
  row to type each cell's value (saved with the scenario and sent on run).
  A Sleep operation can be dragged from the
  Operations panel under 'All Messages' (with a millisecond duration); when the
  running scenario reaches the Sleep step it pauses the timeline for that long
  (every later step is delayed). A Response operation (drag it the same way)
  waits for a chosen message: when it is received and the selected field equals
  the entered value, the configured message is sent automatically. A scenario
  is stored in its own folder as `Configuration.xml` plus one
  `{index}_{message}.json` file per step.
- Record button: captures every incoming and outgoing message to a binary `.bin`
  file. The Record tab opens such a file, lists the message types on the left,
  shows every recorded message of the selected type in the table and decodes
  all the fields of the message clicked in the table. 'Export Type (CSV)' writes
  the selected type to a CSV file: one column per field, one row per recorded
  message (6 received messages of a type produce 6 rows).
- Graph tab: live message timeline with a TX lane (MBT → NIRON) and an RX lane
  (NIRON → MBT), per-message filter checkboxes (each showing the live number of
  sent/received messages of that type), zoom in/out (timeline header
  buttons, View Options or Ctrl+mouse wheel), direction / pause options
  and a details table showing when every message arrived. Clicking a point on the
  timeline focuses its row in the details table and decodes all the fields of that
  message (clicking a details row decodes it as well). Under heavy traffic the
  graph repaints at most ~3 times per second and draws only the newest events,
  so the UI stays responsive.
- Record Graph tab: opens a recording `.bin` file (captured with the Record
  button) and displays it on the same two-lane timeline as the Graph tab, with
  per-type filter checkboxes (each showing how many messages of that type are
  in the file), zoom / direction options, a details table and
  full field decoding; clicking a timeline point focuses its row in the
  details table.
- Statistics tab: a Global Statistics panel (total messages/bytes sent and
  received, current and peak messages/sec and bytes/sec, average traffic rate,
  running time, CPU and memory usage, queue size, connected interfaces, packet
  loss, error / CRC / timeout / retransmission counters and average processing
  time per message) next to a per-message table with totals (received/sent),
  the current rates (messages per second and bytes per second), total bytes,
  average / minimum / maximum message size, last RX/TX timestamps, seconds
  since the last message, receive errors and dropped (unknown id) messages;
  refreshed every second, with a totals summary line and a reset button.
- Flow tab: a visual scenario designer. Drag messages and flow items from the
  left palette onto the canvas - a Message is drawn as a rectangle and a Condition
  as a rhombus (Delay and Loop too). Move a node by dragging it, right-click two
  nodes to connect them (the arrows set the send order and each node shows its
  order number), and edit the selected step in the Step Properties panel. The
  'Run Scenario' button plays the flow: it sends each Message in order, waits on a
  Condition for its message (with timeout), pauses on a Delay and repeats on a Loop.
  A Condition can also carry a logical test on a field of the awaited message
  (e.g. Field1 >= 5, chosen in Step Properties): when the message arrives the field
  is compared and the True branch is taken when it holds, otherwise the False branch.
  Each Condition result can also send a message: pick a 'Send Message' for the True
  and / or False outcome and it is sent (with default field values) before the branch
  is followed.
  A Loop can also select a 'Send Message' that is sent (with default field values)
  on each iteration when the flow reaches the loop.
- Look & Feel selector in the header: Dark Blue (default), Midnight Black,
  Light and Military Green themes; the whole UI (panels, tables, graphs)
  restyles instantly.
- Precision timing: periodic sends (Simulator tab and scenario steps) run on
  dedicated high-priority PrecisionTimer threads with 1 ms Windows timer
  resolution and drift-free absolute scheduling, so intervals as small as
  10 ms are accurate. The receive path records on the transport thread and
  never blocks on the UI, so fast incoming rates stay accurate too.
